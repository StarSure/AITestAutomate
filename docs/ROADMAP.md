# Roadmap

## Phase 0: Product Validation

Duration: 2 weeks

Goal:

Make sure we are solving a real QA regression problem before writing too much code.

Deliverables:

- 5 QA interviews
- 10 real regression flow examples
- MVP scope confirmed
- Demo story confirmed
- README improved

Exit criteria:

- At least 3 QA users say they would try the tool.
- At least 5 regression flows can be represented by Flow Spec.
- The first demo can be explained in under 3 minutes.

## Phase 1: Local Demo

Duration: 3 to 4 weeks

Goal:

Build a local demo that proves the core workflow.

Deliverables:

- Basic web console
- Basic API server
- Flow Spec editor
- Playwright test generation
- Manual run button
- Run result page
- Screenshot and trace collection

Exit criteria:

- A user can create one flow and run it locally.
- A failed run shows screenshot and error message.
- The demo works on a sample web app.

## Phase 2: Community Alpha

Duration: 4 to 6 weeks

Goal:

Make the project usable by early community users.

Deliverables:

- Docker Compose deployment
- Project setup screen
- Environment setup screen
- Run history
- Artifact viewer
- AI failure classification
- Basic documentation
- Example project

Exit criteria:

- A new user can run the demo in less than 15 minutes.
- 3 external users can deploy it.
- 10 common failure cases are classified.

## Phase 3: Community Beta

Duration: 6 to 8 weeks

Goal:

Make the platform useful for small QA teams.

Deliverables:

- Scheduled runs
- GitHub integration
- PR comment summary
- Failure clustering
- Repair suggestion review
- Basic user accounts
- Better onboarding guide

Exit criteria:

- A QA team can use it for weekly regression.
- Failure reports are clear enough for bug reporting.
- The community can contribute issues and examples.

## Phase 4: Community 1.0

Duration: 8 to 10 weeks

Goal:

Release a stable open-source community version.

Deliverables:

- Stable Flow Spec format
- Versioned documentation
- Installer guide
- Contribution guide
- Plugin interface draft
- LLM provider abstraction
- Security notes

Exit criteria:

- The project has a clear public demo.
- The project can be self-hosted reliably.
- Contributors can understand how to add features.
- The core workflow is stable.
